1. test_read_only_connection assertion on journal_mode is tautological
   tests/test_db.py:64  — `assert journal_mode == "wal"` after opening
   the DB read-only. SQLite reports the *file's* current journal_mode
   (still WAL from the migrate call), not the mode this connection just
   set. The test would pass even if `get_connection(read_only=True)`
   silently tried to set WAL on a read-only handle. Suggest a more
   specific assertion (e.g. open against a non-WAL file and confirm
   it stays whatever it was) or a comment explaining the intent.
   Severity: minor — does not block ship; future maintainer trap.

2. ASR_MODEL / ASR_COMPUTE_TYPE set to empty string in prod compose
   docker-compose.prod.yml:20-21
       - ASR_MODEL=${ASR_MODEL:-}
       - ASR_COMPUTE_TYPE=${ASR_COMPUTE_TYPE:-}
   The host-level `.env` defines ASR_MODEL/ASR_COMPUTE_TYPE with sane
   defaults (or unset) — `${VAR:-}` substitutes empty string when
   unset, which `shared/config.py:50,52` correctly treats as "no
   override" (`if asr_model := os.environ.get("ASR_MODEL")`). So the
   empty-string default is intentional and works. Worth a one-line
   comment to make that explicit, or omit the env lines entirely so
   workers inherit the host shell's value (or nothing).
   Severity: minor — works correctly, mildly confusing.

3. set_wal_checkpoint_metrics uses `.set()` (Gauge) — last-writer-wins
   shared/metrics.py:90-93
   If multiple workers run on the same DB and call passive checkpoint
   independently, the gauge reflects whichever checked last. This is
   the standard Prometheus pull model and is acceptable; the plan does
   not require a multi-worker aggregation story for WP-13.
   Severity: minor — note for future multi-worker scaling.

4. WalCheckpointMode typing vs sqlite column typing
   shared/db.py:67-73 — `busy` is declared `bool` and constructed via
   `bool(row[0])`. SQLite returns 0/1 for the busy column; the cast
   is correct. `log_frames`/`checkpointed_frames` use `int(row[N])`.
   Safe today; if sqlite ever returns NULL on error, the `int()`
   cast will raise TypeError — unhandled, would propagate out of
   `passive_wal_checkpoint`'s `except Exception` (good — logged
   and continued) but the gauge won't update. Acceptable per
   plan; not a blocking concern.
   Severity: minor.

5. worker/janitor.py logs debug-level for "wal checkpoint passive busy"
   worker/janitor.py:142-149
   Busy is a normal, frequent condition under concurrent readers;
   operators tailing at INFO will not see it. If the operator wants
   visibility into WAL pressure, change to `logger.info` or document
   the level. Not a defect — judgment call.
   Severity: minor.
